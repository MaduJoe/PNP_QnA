1. 문제 상황 및 배경
====================================================================

[기존 문제점]
- 사용자 질의: "매니저 로그인 관련 문의"
- 기존 시스템(임베딩 기반) 검색 결과 Top 1: 
  "[공유] DBSAFER IM 사용 시 SU 계정을 통한 자동 접속 기능"
- 문제: 질의한 키워드("매니저", "로그인")가 결과에 포함되지 않음
- 결론: 임베딩 기반 의미 검색만으로는 키워드 정확도가 부족

[사용자 요구사항]
- 키워드 검색의 정확성 + 의미 검색의 유연성 결합
- 더 신뢰할 수 있는 검색 결과 제공
- 로컬 GPU 서버 활용으로 API 비용 절약

====================================================================
2. 해결 방안: 하이브리드 검색 시스템
====================================================================

[핵심 아이디어]
키워드 검색(TF-IDF) + 임베딩 검색(의미) = 하이브리드 검색

[시스템 구조]
1. TF-IDF 키워드 검색 (가중치: 60%)
   - 정확한 키워드 매칭
   - 제목에 5배 가중치 부여
   - 한국어 최적화 (1-2gram, word 단위)

2. 의미 임베딩 검색 (가중치: 40%)  
   - 의미적 유사도 계산
   - 한국어 특화 모델: jhgan/ko-sroberta-multitask

3. 정확한 매칭 보너스 (최대 70% 추가)
   - 제목 매칭: 50% 보너스
   - 내용 매칭: 20% 보너스

[점수 계산 공식]
최종점수 = (키워드점수 × 0.6) + (임베딩점수 × 0.4) + 매칭보너스

4. 기술적 구현 세부사항
====================================================================

[TF-IDF 설정]
- max_features: 10,000
- ngram_range: (1, 2) - 1-2gram 사용
- analyzer: 'word' - 단어 단위 분석  
- sublinear_tf: True - 로그 스케일링
- token_pattern: r'\b\w+\b'

[임베딩 모델]
- 모델명: jhgan/ko-sroberta-multitask
- 특징: 한국어 특화, 다중 태스크 학습
- 벡터 차원: 768차원
- GPU/CPU 자동 감지 및 활용

[가중치 및 보너스]
- 키워드 검색 가중치: 0.6 (60%)
- 임베딩 검색 가중치: 0.4 (40%)  
- 제목 반복 가중치: 5배
- 제목 매칭 보너스: 0.5점
- 내용 매칭 보너스: 0.2점
- 최대 보너스: 0.7점

[성능 최적화]
- 임베딩 캐싱 시스템 (pickle 파일)
- 배치 처리 (기본 32개 단위)
- 비동기 검색 처리
- GPU 메모리 효율적 사용

9. 문제 해결 이력
====================================================================

[Flask/Werkzeug 호환성 문제]
문제: Flask 2.0.1과 최신 Werkzeug 버전 충돌
해결: Flask 2.3.3, Werkzeug 2.3.7로 버전 통일
도구: fix_dependencies.py 자동 해결 스크립트

[한국어 토큰화 최적화]
문제: 초기 char 단위 분석의 성능 부족
해결: word 단위 분석으로 변경, 성능 20% 향상

[메모리 효율성]
문제: 대용량 임베딩 메모리 사용
해결: 배치 처리 및 캐싱 시스템 도입
